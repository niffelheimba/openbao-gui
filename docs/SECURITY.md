# Security model

## Trust boundaries

- The initial root is authenticated only by the installer hash delivered through
  a separate trusted channel. SmartScreen's **Run anyway** is not verification.
- Windows Schannel and the Current User root store validate all OpenBao HTTPS
  traffic. There is no insecure TLS mode.
- OpenBao policies and identity-templated PKI roles authorize names and usages.
  UI restrictions are convenience only.
- Kanidm client credentials remain on OpenBao.
- OpenBao tokens exist only in Rust memory, never browser storage or disk.
- Private keys are generated by Microsoft Software Key Storage Provider as
  non-exportable Current User CNG keys.

## Failure behavior

- Configuration and embedded fingerprint mismatches stop startup.
- Authentication polls have a five-minute deadline and support cancellation.
- API errors are reduced to bounded, redacted user messages.
- A failed issuance deletes its named CNG key container.
- Certificate identity, key match, validity, and required EKUs are checked before
  Windows accepts the response.

## Known v1 limitations

- The client has no unattended renewal because sessions are memory-only.
- Root rollover must happen while the old HTTPS path remains trusted or through a
  newly hash-verified installer.
- The application is unsigned and therefore depends on the documented checksum
  ceremony.
- Certificate export and machine-wide trust are intentionally unsupported.

Report vulnerabilities privately to the repository owner. Do not include tokens,
private keys, full CSRs, or identity claims in reports.
